# Ghostline Demo Mode

## Purpose

Demo Mode exists to make Ghostline **filmable, repeatable, and judge-friendly**.

This project is being optimized for a polished, under-4-minute recorded demo.  
That means Ghostline must not rely on chance, loose improvisation, or environment-dependent randomness during recording.

Demo Mode is a **first-class execution system**, not optional polish.

Its job is to guarantee:

- one reliable story path
- one reliable interruption moment
- one reliable recovery moment
- one reliable final report
- one reliable presentation flow

---

## Demo Mode Philosophy

Normal Ghostline mode can be adaptive and somewhat flexible.

Demo Mode must be:

- deterministic
- safe
- repeatable
- readable
- emotionally coherent
- easy to rehearse
- easy to reset between takes

Demo Mode should not change the core product fantasy.  
It should simply **lock the best path** through it.

---

## Demo Goals

The demo must prove all of these clearly:

1. It is a **Live Agents** project
2. The operator asks for camera access **in-call**
3. The user and operator interact in real time
4. The room is part of the gameplay
5. There is a visible grounding HUD
6. There is a genuine **Ready to Verify** flow
7. There is one clear **barge-in interruption** moment
8. There is one clear **recovery/failure** moment
9. The operator feels alive between tasks
10. The experience ends with a case report
11. The project is cloud-hosted and real

Demo Mode should optimize for this proof, not for open-ended play.

---

## Demo Path Requirements

Demo Mode must use a **fixed safe task sequence**.

It should avoid fragile optional-object dependencies unless they are intentionally staged and extremely reliable.

### Recommended Demo Task Sequence

This is the safest baseline path:

1. **Show Threshold**
2. **Close Boundary** or stable boundary acknowledgment
3. **Increase Illumination**
4. **Stabilize Camera / Hold Still**
5. **Paper Anchor** or **Clear-Surface Anchor**
6. **Containment Phrase / Seal**

This path is preferred because it relies on:
- threshold
- light
- stillness
- flat surface
- speech

These are broadly available and easy to stage on camera.

### Optional Demo Flavor Step

A simple mark on paper can appear if it is stable and easy to demo, but it should not become a fragile gate.

---

## Demo Narrative Spine

Demo Mode should follow one consistent story arc:

### 1. Hotline Pickup
Operator answers the call and frames the situation.

### 2. In-Call Camera Request
Operator requests camera access naturally.

### 3. Room Assessment
User shows the room. Operator chooses the path.

### 4. Boundary Securing
Operator establishes the room’s threshold or entry point.

### 5. Verification Success
One early success proves grounding.

### 6. Diagnosis Beat
Operator asks a short question and classifies the incident.

### 7. Barge-In Moment
User interrupts; operator stops and restates.

### 8. Controlled Near-Failure
One verification or framing issue fails in a believable way.

### 9. Recovery Beat
Operator gives a clear corrective instruction; verification succeeds.

### 10. Anchor / Seal
Operator completes final action and phrase.

### 11. Case Report
Show the report with confirmed vs user-confirmed-only outcomes.

This narrative should feel like one coherent incident response.

---

## Demo Timing Target

The full demo should target roughly **3:40–3:55** total runtime.

### Suggested Beat Timing

- **0:00–0:20** Hook and product framing
- **0:20–0:40** Call pickup + camera request
- **0:40–1:05** Room scan + calibration + HUD visible
- **1:05–1:35** First task and first verification
- **1:35–1:55** Diagnosis beat
- **1:55–2:20** Barge-in moment
- **2:20–2:55** Controlled failure + recovery
- **2:55–3:20** Final anchor / phrase / closure
- **3:20–3:45** Case report
- **3:45–3:55** Quick architecture / cloud reminder if needed

Stay under 4 minutes.

---

## Demo Mode Determinism Rules

When Demo Mode is enabled:

- use fixed task order
- use fixed diagnosis beat
- use fixed or highly constrained flavor lines
- use fixed sound cue timing
- use one fixed interruption point
- use one fixed recovery point
- suppress random variation unless it is strictly bounded and harmless

Demo Mode must prefer polish over variety.

---

## Demo-Specific Operator Dialogue Rules

Demo Mode should use a **curated set of best lines**, not the full rotating library.

This includes:

- opening line
- camera request line
- first task line
- waiting-state lines
- diagnosis line
- interruption restatement line
- recovery line
- final closure line

The operator should still sound like the real product, but the wording should be controlled and repeatable.

### Good Demo Dialogue Characteristics

- short
- calm
- clear
- procedural
- readable in subtitles
- emotionally coherent
- not too wordy

### Avoid in Demo Mode

- too much variation
- extra lore
- rambling diagnosis
- repeated lines
- lines that are hard to subtitle cleanly
- jokes or tonal inconsistency

---

## Demo Diagnosis Beat

Demo Mode must include exactly one clear diagnosis beat.

That beat should:
- come between major tasks
- feel like an operator assessing the case
- add realism without slowing the demo
- support the classification label shown later

### Example diagnosis beat structure

1. Operator asks: “What did the sound resemble?”
2. User answers briefly.
3. Operator says something like: “That matches threshold activity.”

This is enough.  
Do not let diagnosis become a long branching scene during the recording.

---

## Demo Barge-In Moment

Barge-in is a centerpiece and should happen **once**, clearly, and on cue.

### Requirements

- it must interrupt a meaningful operator line
- operator audio must stop immediately
- queued audio must be flushed
- HUD must visibly reflect interruption
- transcript must show the interruption
- operator must restate the instruction more briefly afterward

### Demo scripting requirement

This moment should be staged and rehearsed enough that it works on the first or second take, not as a probabilistic event.

---

## Demo Near-Failure and Recovery Beat

Demo Mode must include one controlled failure moment.

This should be a **safe, repeatable, believable** failure such as:

- slight blur
- slight framing issue
- low-light moment
- missing enough of the frame to verify

### Requirements

- the first verification should fail honestly
- the HUD should show unconfirmed status and block reason
- the operator should issue one recovery line
- the user should correct the issue
- the second attempt should succeed

The recovery beat should prove:
- honesty
- visible grounding
- controlled adaptation

It must not create cascading instability in the rest of the demo.

---

## Demo HUD Requirements

The HUD is especially important during recording.

Demo Mode should ensure the HUD is:

- always visible
- easy to read on camera
- showing meaningful state changes
- not overcrowded
- aligned with the actual session state

If needed, Demo Mode may apply a **demo-friendly HUD treatment** such as:
- slightly larger text
- stronger contrast
- simplified spacing

But the HUD content must remain faithful to the real product.

---

## Demo Sound Rules

Demo Mode should use a carefully controlled sound sequence.

### Requirements

- subtle ambient bed
- one or two clean stingers
- one escalation cue
- one final containment result cue
- ambient ducking during operator speech

### Important rule

Sound must support:
- tension
- pacing
- transitions

Sound must **not**:
- obscure voice
- obscure subtitles
- feel like a trailer
- create clutter during the interruption moment

Demo-specific sound timing may override the general event rules if needed for polish.

---

## Demo Reset and Rehearsal Requirements

Demo Mode must support fast repeatability.

### Required features

- quick reset between takes
- clear return to initial state
- no stale transcripts, old HUD state, or leftover media tracks
- no leftover Gemini session state
- no stale report state

### Rehearsal support

There should be a way to repeatedly test:
- the fixed task sequence
- the interruption moment
- the recovery moment
- the final report

This can be a lightweight rehearsal harness, a debug route, or a demo reset control.  
It does not need to be a complex QA system.

---

## Demo Report Requirements

The final report shown in the demo should clearly communicate:

- case ID
- classification label
- completed tasks
- which tasks were:
  - confirmed
  - user-confirmed-only
  - unverified or skipped
- final verdict:
  - secured
  - partial
  - inconclusive

The report should be visually strong enough to act as the emotional payoff of the demo.

If alternate report templates exist, Demo Mode should use the template that best fits the staged outcome.

---

## Demo Constraints

Demo Mode must not introduce:

- different product fantasy
- extra lore trees
- new mechanics not in the main product
- fake verification presented as real
- unstable random task selection
- noisy sound clutter
- excessive operator dialogue

Demo Mode is not a separate game.  
It is the **best presentation path** through Ghostline.

---

## Demo Success Criteria

Demo Mode is successful if it consistently produces:

- a strong opening
- a clear in-call permission moment
- visible camera-aware behavior
- visible grounding HUD
- a believable task progression
- one diagnosis beat
- one interruption beat
- one recovery beat
- one strong final report
- a path that can be recorded repeatedly without breaking

That is the standard.
